 Standaufzeichnung + Roadmap: NATASCHA-Port „next level"

 Context

 Der native NATASCHA-Port in die LUA-UI ist weit gediehen (Korrektur, Klassen,
 Schüler, Aufgabe/Rubrik anlegen, Erwartungshorizont-Generator). Der User will:
 (1) eine präzise Standaufzeichnung (was von NATASCHA ist migriert?), (2) den
 Erwartungshorizont-Creator vertiefen, (3) eine konkrete UX-Kleinigkeit (Baukasten:
 ganzen Blocktyp per X entfernen), (4) weitere „Kleinigkeiten" + Lücken, (5) Ideen,
 wie die App „next level" wird (u. a. Korrektur-Vorschau nach Generierung).

     ---
     TEIL C — NATASCHA-Parität schließen

     - Erwartungshorizont vertiefen (der „komplexe" Punkt): „Bearbeiten" (Textarea) +
     „Akzeptieren & speichern" → CLI-Erweiterung nutzt vorhandenes
     nc.save_erwartungshorizont_to_config (natascha_core.py:1609) → schreibt
     rubrics/erwartungshorizont_*.md + Config-Link; Korrektur nutzt ihn dann
     automatisch (cmd_analyze übergibt schon erwartungshorizont_name). + Provider/
     Modell-Auswahl in der View.
     - LLM Klassen-Briefing & Schüler-Profil zu Ende bauen: CLI liefert heute nur den
     Prompt → LLM-Call + Save (save_klassen_briefing/save_schueler_profil existieren)
       - Anzeige in KlassenView/SchuelerView. Hoher didaktischer Wert.
     - Korrektur-Feedback-Vorschau in-app (User-Frage „Vorschau nach Generierung?"):
     markierten Schülertext + Kriterien + Fehler als A4-Vorschau rendern (LUA hat bereits
     einen DIN-A4-Vorschau-Container für Arbeitsblätter — Muster wiederverwenden), statt
     nur DOCX auf Platte. → klares „ja, machen".
     - Batch-Korrektur: ganze Klasse/Ordner auf einmal, Fortschritts-Events
     (Tauri-Event-Stream) + Abbrechen.
     - Rubrik-Editor, Schüler-CSV-Import, Retro-Import (kleinere, klar umrissene Stücke).

     ---
     TEIL D — „Next Level" (über Parität hinaus)

     1. DER Closed Loop, jetzt in-app (Killer-Feature): In KlassenView/Heatmap ein
     Button „Übungsblatt zu Top-Fehlern generieren" → springt direkt in LUAs
     Generator (Step0) mit vorbefüllten meta.fokusThemen (Mechanik existiert aus
     Phase 1!). Da jetzt eine App + gemeinsame DB, braucht es die Datei-Brücke gar
     nicht mehr — ein Klick von „korrigiert" zu „passendes Übungsblatt". Das ist der
     eigentliche Produkt-Sinn.
     2. Differenzierte Übungen pro Schüler: aus Längsschnitt/Profil je Schüler ein
     Übungsblatt zu seinen Schwächen (nutzt fokusThemen + Bloom-Gating).
     3. Start-/Dashboard-Ansicht: Überblick (zuletzt korrigiert, Klassen mit offenem
     Handlungsbedarf, Lehrplan-Abdeckung).
     4. Erwartungshorizont & Rubrik als editierbare, versionierte Objekte (statt loser
     Dateien) — Grundlage für konsistente Bewertung.

     ---
     Reihenfolge (User wählte ALLE vier Thrusts — volle Roadmap)

     1. Quick Wins (Teil B) — sofort, klein, sichtbar (Baukasten-X zuerst).
     2. Der Closed Loop (D1) — größter Hebel, baut auf Vorhandenem auf.
     3. Erwartungshorizont-Save + Korrektur-Vorschau (Teil C) — die „komplexen" Themen.
     4. LLM-Briefing/Profil + Batch-Korrektur — Parität & Workflow.

     Jede Stufe wird einzeln gebaut, verifiziert (siehe unten) und committet, bevor die
     nächste beginnt — kein Big-Bang.

     Verifikation (pro Schritt)

     pnpm --filter @lehrunterlagen/web typecheck && pnpm -r test · cargo check/cargo test
     · bei CLI-Änderungen py_compile + CLI-Smoke · GUI-Roundtrip auf Windows
     (pnpm tauri:dev). Closed-Loop-Beweis: Heatmap → „Übungsblatt generieren" → Step0 mit
     fokusThemen vorbefüllt → DOCX zu den Top-Fehlern.
